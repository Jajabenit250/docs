## Design principles

- **Codes, not UUIDs.** Every payload uses human-readable codes (`fosaId`, `practitionerLicenseNumber`, `productCode`, ICD-11 `diagnosisCode`, `nationalId`, `beneficiaryNumber`).
- **Multiple alternate keys.** Facilities by FOSA, Minisante matricule, supplier, or TIN. Practitioners by licence, alt-licence, NID, passport, or phone. Products by product code, alt-code, ATC, or name.
- **One `scheme` field.** `"CBHI"` or `"RAMA"` on every multi-scheme request. The gateway enforces scope against your JWT.
- **Unified error envelope.** `{ error: { code, message, details[] } }` with stable codes across both schemes.
- **Idempotent writes.** Safe to retry — pass `X-Idempotency-Key` and we'll return the cached result for 24 hours.

## Quickstart

Three requests and you've exercised auth, lookup, and eligibility.

<Steps>
  <Step title="Authenticate">
    Exchange a service secret for a Bearer token.
    ```bash
    curl -X POST http://localhost:4010/api/v1/partner/session \
      -H 'Content-Type: application/json' \
      -d '{"serviceName":"moh-eubuzima","serviceSecret":"demo-moh-secret"}'
    ```
  </Step>
  <Step title="Look up a facility">
    By any of its codes.
    ```bash
    curl http://localhost:4010/api/v1/partner/facilities/MF00632 \
      -H "Authorization: Bearer $TOKEN"
    ```
  </Step>
  <Step title="Check CBHI eligibility">
    For a household.
    ```bash
    curl -X POST http://localhost:4010/api/v1/partner/eligibility/check \
      -H "Authorization: Bearer $TOKEN" -H 'Content-Type: application/json' \
      -d '{"scheme":"CBHI","householdHeadNationalId":"1197080098765432"}'
    ```
  </Step>
</Steps>

<Tip>
  **Prefer to click?** Every endpoint is live in the [API Reference](/api-reference/introduction) — pick an operation, edit parameters, hit **Send**.
</Tip>
