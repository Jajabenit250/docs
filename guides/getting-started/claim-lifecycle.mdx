---
title: "Claim lifecycle"
description: "A patient visit flows through a known set of states from submission to adjudication. Partners can pull the timeline any time, resubmit after rejection, and close out inpatient episodes."
---

<Note>
  **The happy path** ‚Äî Visit recorded ‚Üí (optional) `PENDING_MEDICAL_ADVISOR_REVIEW` ‚Üí status updates via webhook.

  In states: `SUBMITTED` ‚Üí `UNDER_REVIEW` ‚Üí `APPROVED`. Unhappy fork: `REJECTED` ‚Üí partner resubmits ‚Üí back to `UNDER_REVIEW`.
</Note>

## States

| State | Meaning |
| --- | --- |
| `SUBMITTED` | Visit created via `POST /patient-visits`. Queued for adjudication. |
| `UNDER_REVIEW` | Medical advisor reviewing. No partner action needed. |
| `APPROVED` | Adjudication passed. Terminal success state. |
| `REJECTED` | Adjudication failed. See `statusHistory[].reasonCode`; resubmit via `POST /patient-visits/{ref}/resubmit`. |
| `RESUBMITTED` | Partner submitted corrections; re-enters adjudication. |
| `DISCHARGED` | Inpatient visit closed (re-computes bed-day billing). |

## Inspect the timeline

Every state change is appended as a `StatusHistoryEntry` with `actor`, optional `reasonCode`, and `comment`.

```http
GET /api/v1/partner/patient-visits/VST-20260420-0001/status-history

{
  "visitReference": "VST-20260420-0001",
  "statusHistory": [
    { "code": "SUBMITTED",    "actor": "chuk-emr",       "occurredAt": "2026-04-20T09:12:14Z" },
    { "code": "UNDER_REVIEW", "actor": "rama-advisor",   "occurredAt": "2026-04-20T11:00:00Z" },
    { "code": "REJECTED",     "actor": "rama-advisor",
      "reasonCode": "MISSING_SUPPORTING_DOCUMENT",
      "comment": "Attach discharge summary", "occurredAt": "2026-04-21T08:30:00Z" }
  ]
}
```

## Rejection reason codes

Stable enum values from `/reason-codes`. Safe to branch on in code.

- `INELIGIBLE_MEMBER` ‚Äî coverage lapsed at service date.
- `MISSING_SUPPORTING_DOCUMENT` ‚Äî required attachment not provided.
- `DUPLICATE_CLAIM` ‚Äî overlapping visit already on file.
- `TARIFF_EXCEEDED` ‚Äî posted price above catalogue tariff.
- `UNAUTHORIZED_PRODUCT` ‚Äî product needs pre-authorization.
- `PRACTITIONER_NOT_LICENSED` ‚Äî practitioner licence inactive at facility.
- `INVALID_REFERRAL` ‚Äî higher-level care without a valid referral.
- `DIAGNOSIS_PRODUCT_MISMATCH` ‚Äî product not clinically indicated.
- `QUANTITY_EXCEEDS_LIMIT` ‚Äî posted quantity exceeds rule-book limit.
- `OUT_OF_STOCK` ‚Äî dispensing not confirmed.
- `OTHER` ‚Äî free-text reason, `comment` required.

## Resubmit a rejected visit

<Steps>
  <Step title="Fetch the rejection reason">
    Either via webhook (`*.patient-visit.rejected`) or by polling `/status-history`. Read `reasonCode` and `comment`.
  </Step>

  <Step title="Fix the underlying data">
    Attach the missing document, correct the tariff, obtain the referral ‚Äî whatever `reasonCode` demanded. Do this server-side before calling the API.
  </Step>

  <Step title="Call /resubmit with the correction">
    ```http
    POST /api/v1/partner/patient-visits/VST-20260420-0001/resubmit
    {
      "reasonForResubmission": "Added discharge summary as requested",
      "supportingDocuments": [
        { "type": "DISCHARGE_SUMMARY", "url": "https://files.example.rw/ds-0001.pdf" }
      ]
    }
    ```

    Only valid while the visit is `REJECTED`. The backend appends a `RESUBMITTED` entry and re-queues the visit.
  </Step>
</Steps>

## Discharge inpatient visits

```http
POST /api/v1/partner/patient-visits/VST-20260418-0033/discharge
{ "dischargeDate": "2026-04-21" }
```

Only valid when `healthCareStayType=IN_PATIENT`. If you omit `daysStayed`, the backend computes it from admission ‚Üí discharge.

<Tip>**Inpatient cron.** RAMA auto-discharges inpatient visits with no activity for 30 days. If you want an explicit discharge date on the record, call `/discharge` yourself.</Tip>
